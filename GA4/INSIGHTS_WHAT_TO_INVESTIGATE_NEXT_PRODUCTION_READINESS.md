# GA4 Insights What To Investigate Next Production Readiness

## Purpose

This file tracks the planned hardening work for the GA4 Insights `What to investigate next` section.

The section should remain an executive-facing investigation guide, not a causal diagnosis engine.

## Root Cause Summary

The current section is generated by one frontend rule engine and rendered as one flat severity-sorted list.

Current risk:

- setup issues, KPI/Benchmark status, data-quality warnings, anomaly signals, and generic recommendations are mixed together
- recommendations can read as causal even when they are only rule-based checks
- the section does not clearly show the data basis behind each card
- impossible saved targets, such as a percentage benchmark above `100%`, can be reported as normal behind-target performance instead of being flagged as invalid configuration
- the intro mentions 7-day comparison even when the campaign has too little history for 7-day analysis

## Target Behavior

The section should be:

- deterministic and explainable
- grouped by finding type
- explicit about data basis and history sufficiency
- conservative about causal wording
- able to flag invalid KPI/Benchmark configuration before presenting performance conclusions
- suitable for marketing executives as prioritized investigation guidance

## Commit Plan

### Commit 1: Group Findings By Type

Status: Implemented and locally validated

Goal:

- separate findings into clear groups instead of one flat list

Proposed groups:

- `Data setup issues`
- `Targets off track`
- `Trend signals`
- `Revenue and spend checks`
- `Informational context`

Smallest safe implementation:

- add a local `category` field to generated Insight items
- render grouped cards in the existing section
- preserve existing card content and severity logic
- keep generated report output compatible with the existing Insights section

Validation:

- local regression guard added for group labels and grouped render path
- passed `npm test -- --run server/ga4-insights-regression.test.ts`
- passed `npm test -- --run server/ga4-ui-regression.test.ts`
- passed `npm run check`
- confirm missing spend appears under setup issues
- confirm behind KPI/Benchmark items appear under targets
- confirm GA4/imported revenue double-count warning appears under revenue and spend checks
- confirm empty-state behavior still works

### Commit 2: Add Invalid KPI/Benchmark Configuration Detection

Status: Not started

Goal:

- stop impossible saved targets from appearing as normal performance findings

Smallest safe implementation:

- detect invalid saved KPI/Benchmark values in the Insights engine
- show a high-priority configuration card for invalid values
- skip normal behind-target/behind-benchmark conclusions for invalid items
- do not rewrite existing saved data automatically

Validation:

- percent metrics such as `Engagement Rate` and `Conversion Rate` flag targets above `100%`
- count, currency, ratio, and percent units reject negative values where not meaningful
- invalid items do not create misleading behind-target cards
- valid custom KPI/Benchmark values still calculate normally

### Commit 3: Add Data Basis And Confidence Labels

Status: Not started

Goal:

- make each card clear about what data it is based on and how strong the signal is

Smallest safe implementation:

- add display-only metadata to Insight cards:
  - data basis, such as `GA4 completed daily history`, `GA4 to-date totals`, `Imported revenue`, `Saved KPI`, or `Saved Benchmark`
  - confidence, such as `High`, `Medium`, or `Low`
- do not change metric calculations

Validation:

- setup/config cards show high confidence
- trend/anomaly cards show confidence based on available history
- KPI/Benchmark cards identify saved target inputs
- recommendations remain readable on desktop and mobile

### Commit 4: Make Trend Copy History-Aware

Status: Not started

Goal:

- avoid implying 7-day analysis exists before enough daily history is available

Smallest safe implementation:

- make the section intro conditional:
  - fewer than 6 days: explain anomaly detection needs more history
  - 6 to 13 days: explain short-window checks are available
  - 14 or more days: explain 7-day vs prior 7-day checks are active
- preserve existing trend thresholds

Validation:

- with 2 completed days, the intro does not lead with 7-day comparison
- with 6 completed days, short-window copy appears
- with 14 completed days, 7-day comparison copy appears

### Commit 5: Make Recommendations Explicitly Non-Causal

Status: Not started

Goal:

- keep executive trust by wording recommendations as checks, not proven root causes

Smallest safe implementation:

- audit recommendation strings in `What to investigate next`
- replace causal wording with investigation wording where needed
- keep the existing `Recommended check:` label

Validation:

- recommendations say what to check first
- recommendations do not claim a root cause unless the rule directly proves it
- report output uses the same wording

### Commit 6: Regression Coverage And Documentation

Status: Not started

Goal:

- lock the intended behavior with focused regression tests and docs

Smallest safe implementation:

- add focused UI/regression coverage for:
  - grouped rendering
  - invalid target handling
  - history-aware intro copy
  - report output compatibility where applicable
- update `GA4/INSIGHTS.md` after validation

Validation:

- run focused GA4 Insights tests

## Tracking Notes

- Do not change analytics calculations unless a later root cause proves a calculation bug.
- Do not automatically rewrite existing KPI/Benchmark records.
- Treat invalid existing records as configuration issues first.
- Keep response shapes backward compatible.
- Keep changes inside the existing GA4 Insights path unless a specific traced bug requires otherwise.
